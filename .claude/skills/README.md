# Claude Skills for Acoustic Connect Cordova Plugin

This directory contains skills that guide implementation work for the
`cordova-acoustic-connect` Acoustic Connect Cordova plugin.

## Available Skills

### Implementation Skill

**Path**: `.claude/skills/implementation/`

**Use when**: Implementing JIRA tickets (features, bugs, improvements)

**What it does**:

- Guides full ticket implementation workflow
- Enforces TDD: write tests before implementation
- For bugs: requires a failing test that reproduces the issue before any fix
- Validates code quality (TypeScript, ESLint, unit tests, Cordova build)
- Creates pull requests with proper naming

**Key features**:

- TodoWrite-based implementation planning
- Validation script integration (`validate.sh`)
- npm and Cordova CLI workflow guidance

### Unit Test Creation Skill

**Path**: `.claude/skills/unit-test/`

**Use when**: Adding test coverage for new features or bug fixes

**What it does**:

- Guides Jest unit test creation for the plugin TypeScript source (`src/`)
- Unit tests go in `src/__tests__/` (pure logic, no device needed)
- Enforces TDD for new features (write test first, then implement)
- For bug fixes: creates a failing test to confirm the hypothesis before
  the fix is applied, then verifies it passes afterwards

**Key features**:

- Test file placement and naming conventions
- `beforeEach`/`afterEach` lifecycle and `jest.mock` patterns
- Async test patterns (`async/await`, `jest.fn`)
- Coverage requirements

### Fix Failing Test Skill

**Path**: `.claude/skills/fix-failing-test/`

**Use when**: Autonomously iterating on a fix for a failing test

**What it does**:

- Runs the failing test via `npm test -- --testNamePattern` to capture the
  exact error
- Analyzes error and relevant source code
- Implements a fix and re-runs the test
- Iterates up to 5 attempts, documenting each try
- Determines if the issue is in test code or plugin code

### Diagnose Test Results Skill

**Path**: `.claude/skills/diagnose-test-results/`

**Use when**: Analyzing test failures from Jest or CI output

**What it does**:

- Analyzes Jest output and coverage reports
- Identifies failing tests in the plugin source
- Provides debugging guidance and root cause analysis
- Covers module resolution errors, assertion errors, async issues,
  and TypeScript/build failures

## Project Structure

```
cordova-acoustic-connect/
├── src/                    # TypeScript plugin source
│   └── __tests__/          # Unit tests (Jest, no device)
├── www/                    # Compiled JS output (plugin public API)
├── plugin.xml              # Cordova plugin manifest
├── package.json            # Plugin metadata + scripts
├── tsconfig.json           # TypeScript config
├── applications/
│   └── Demo/               # Demo Cordova app (Android + iOS)
├── docs/                   # Project documentation
└── Jenkinsfile             # CI pipeline (Jenkins, macOS agent)
```

## Test Locations

| Type | Path | Notes |
|---|---|---|
| Unit tests | `src/__tests__/` | Jest, JVM only, no device |
| Integration/E2E | `applications/Demo/` | Requires device/emulator via Cordova |

### Test file naming

Test files mirror the plugin source path:

- Plugin source: `src/AcousticConnect.ts`
- Test file: `src/__tests__/AcousticConnect.test.ts`

## Skill Usage Patterns

### Starting a New Feature Ticket

1. Use **Implementation Skill** to start ticket
2. Use **Unit Test Creation Skill** first (TDD) — write tests before
   implementing the feature
3. Implement the feature in TypeScript to make the tests pass
4. Run `validate.sh` before creating PR

### Fixing a Bug

1. Use **Implementation Skill** to start ticket
2. Use **Unit Test Creation Skill** to write a failing test that reproduces
   the bug — do not touch the fix yet
3. Confirm the test fails for the right reason
4. Implement the fix; confirm the test now passes
5. Run `validate.sh` before creating PR

### Fixing a Failing Test

1. Use **Fix Failing Test Skill** with the failing test name
2. Use **Diagnose Test Results Skill** if the failure pattern is unclear

## Skill Conventions

All skills follow these patterns:

- `SKILL.md` — main skill documentation
- Clear "When to Use" section
- Step-by-step instructions
- Code examples where applicable
- Checklists for verification

## Language Guidelines

These conventions apply across all skills:

- **Plugin source** (`src/`): TypeScript. All new source files in TypeScript.
- **www/**: Compiled output — do not edit directly; generated by `tsc`.
- **Tests**: Jest with `ts-jest`. Files named `*.test.ts`.
- **Null safety**: Use optional chaining (`?.`) and nullish coalescing (`??`).
  Avoid non-null assertions (`!`) unless unavoidable.
- **Async**: Prefer `async/await` over raw Promise chains.

## Adding New Skills

2. Add `SKILL.md` with standard sections
3. Include clear "When to Use" guidance
4. Provide step-by-step instructions
5. Add examples and patterns
6. Update this README
